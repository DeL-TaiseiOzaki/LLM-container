# GPU キュー 使い方ガイド

## これは何？

複数人が同じ GPU サーバーを使うため、**順番待ちシステム（キュー）** を導入した。

`q` コマンドでジョブを投げると：
- 空き GPU があれば → **すぐに実行**
- 空きがなければ → **自動で待機**し、空き次第実行

GPU の空き確認・Notion DB への手動登録・`CUDA_VISIBLE_DEVICES` の設定は**不要**になった。

---

## 基本の使い方

### ジョブを投げる

```bash
q -G 1 python train.py
```

- `-G 1` = GPU を 1 枚使う（2 枚なら `-G 2`）
- どの GPU を使うかは**自動で決まる**（自分で指定しなくてよい）
- **ターミナルを閉じてもジョブは止まらない**（SLURM の sbatch と同じ動作）
- ジョブ投入後すぐにプロンプトを返したい場合は `&` をつける

```bash
q -G 1 python train.py &
```

## キューの確認

```bash
q
```

実行すると以下のような一覧が表示される：

```
ID   State      Output           E-Level  Time    GPUs  Command [run=2/8]
3    running    /cache/ts-queue...          12.3s   1     [ozaki] python train.py
4    queued     -                -        -       1     [yamada] python eval.py
5    finished   /cache/ts-queue... 0        45.1s   2     [tagai] python pretrain.py
```

| 項目 | 意味 |
|------|------|
| `ID` | ジョブの番号 |
| `State` | `queued`=待機中 / `running`=実行中 / `finished`=完了 |
| `E-Level` | 終了コード。`0` = 正常終了、それ以外 = エラー |
| `GPUs` | 使用 GPU 数 |
| `[run=2/8]` | 8GPU 中 2GPU が現在使用中 |
| `[ozaki]` | ジョブを投げたユーザー名 |

## 出力（ログ）を確認する

### ジョブの出力を見る

```bash
ts -c 3     # ID=3 のジョブの出力をターミナルに表示（完了後）
ts -o 3     # ID=3 の出力ファイルのパスを表示
```

### リアルタイムでログを追う（実行中のジョブ）

```bash
tail -f $(ts -o 3)    # ID=3 の出力をリアルタイム表示
```

### ログファイルの場所

ジョブの出力は自動的に `/cache/ts-queue/logs/` にファイルとして保存される（nohup と同様）。
手動で確認する場合：

```bash
ls /cache/ts-queue/logs/      # ログファイル一覧
cat $(ts -o 3)          # ID=3 のログファイルを直接読む
```

---

## ジョブを止める・消す

```bash
ts -k 3     # ID=3 のジョブを中止（待機中・実行中どちらも可）
ts -C       # 完了済みジョブを一覧から消去
```

## 優先度・急ぎのジョブ

### キューの先頭に割り込む（urgent）

```bash
q -G 1 --urgent python train.py
```

`--urgent` をつけるとキューの先頭に挿入される。次に GPU が空いたとき最優先で実行される。

### クイックに確認だけしたい場合

短時間（数秒〜数分）の動作確認は `q` 経由で投げれば、キューが空いていればすぐに実行される。
GPU が全部埋まっている場合は `--urgent` で先頭に割り込むか、Slack で相談する。

## CUDA_VISIBLE_DEVICES との併用について

### 基本方針: q コマンドを使う

`q` コマンドを使えば `CUDA_VISIBLE_DEVICES` は自動で設定される。手動指定は**不要**。

### 手動で CUDA_VISIBLE_DEVICES を使いたい場合

`q` を使わずに `CUDA_VISIBLE_DEVICES=3 python train.py` のように直接実行することも**技術的には可能**。
ただし以下のリスクがある：

- **衝突**: キューが同じ GPU を別のジョブに割り当てる可能性がある（キューは手動占有を認識できない）
- **見えない**: `q` の一覧に表示されないため、他のメンバーが GPU の使用状況を把握できない

**やむを得ず手動で使う場合のルール**：

1. `nvidia-smi` で空き GPU を確認する
2. Slack に「GPU X を手動で使います（〜時まで）」と共有する
3. 終わったら Slack で報告する

---

## ルール

- **GPU7 は基本使わない**（他 PJ と共有。他が全部埋まっているときのみ OK）
- 長時間ジョブは事前に Slack で共有する
- 緊急ジョブは Slack で上長・関係者にメンション
- ゾンビプロセスを発見したら上長に連絡
- **手動 CUDA_VISIBLE_DEVICES は原則禁止**（使う場合は上記ルールを厳守）
